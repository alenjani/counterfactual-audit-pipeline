## 1. Executive summary

The repository is in good structural shape. All planned subsystems exist
(generator, validator, auditors, analysis, viz), the engineering principles
in `CLAUDE.md` correctly enforce the quality choices we agreed on (pose
ControlNet, PuLID required, NF4 floor, 1024² resolution, manifest-based
reproducibility), and the statistical plan in `src/cap/analysis/` matches
the OSF pre-registration's H1–H5.

The MVP run completed end-to-end — pipeline plumbing works, Ray actors run,
artifacts are written to the Volume and packaged for HF, the analysis CLI
emits the expected hypothesis-test outputs, and a publication-quality figure
renders. That is a real engineering milestone.

But one number in `results/mvp_demo/analysis/summary.json` invalidates the
audit signal that the pipeline produced:

```json
"identity": {
  "mean_cosine_sim": -0.0036,
  "median_cosine_sim": -0.0066,
  "preserved_fraction": 0.0,
  "n": 600
}
```

`preserved_fraction = 0.0` means **none** of the 600 generated images have
ArcFace cosine similarity to their seed identity above the 0.5 threshold,
and the mean cosine similarity is statistically indistinguishable from
zero (orthogonal embeddings). In ArcFace embedding space, "near zero
cosine sim to the seed" is what unrelated faces look like.

The pre-registration ([preregistration.md, H5](../../research_papers/drafts/isr-counterfactual-audit/preregistration.md))
explicitly makes ArcFace identity preservation ≥ 80% a *manipulation
check that gates H1–H4*: if H5 fails, the confirmatory tests do not run.
At 0.0% preserved, the MVP fails this check by every margin.

The good news: the rest of the architecture is sound, so the fix is local
to the generator subsystem. This review documents what is right, what is
wrong, and what to do next.

### Reusability for P2, P3, P4

The architecture genuinely supports the multi-paper economics we agreed on:

- **P2 — emotion bias as IS governance.** Emotion endpoints are in
  `auditors/aws_rekognition.py`, `azure_face.py`, `face_plus_plus.py`,
  `deepface_local.py`. Adding `emotion` to `tasks:` in a new config and
  re-running `cap-audit` against the same generated images will produce
  a P2-ready dataset with no generator rerun. Pre-registration is the
  only thing standing between MVP and a P2 audit run.
- **P3 — commercial vs OSS sourcing.** The auditor registry's commercial
  tier (Rekognition, Azure, Face++, Google) and OSS tier (DeepFace,
  InsightFace, ArcFace) is exactly the contrast P3 needs. Once API keys
  are provisioned, one run produces the data for both ISR (P1) and the
  P3 sourcing analysis.
- **P4 — longitudinal re-audit.** The `RunManifest` records auditee API
  versions and timestamps where available, which is what P4 needs to
  detect drift at T+18 months. The reproducibility infrastructure is in
  place. Recommend adding a `system_signature` field that hashes the
  auditor's full response schema so regressions in vendor APIs are
  detectable even when version strings don't change.

### Master-plan gaps not yet addressed

1. **Auditee API access** is still listed as "not yet provisioned" in the
   master plan. MVP runs DeepFace only — adequate for plumbing, but the
   ISR submission needs at least the four commercial systems

   contribution credible.
2. **Negative control system** (audit a known-fair-ish system to defend
   against "your method just confirms what you expected") not picked.
   Strong candidate: a face *detection* (not classification) endpoint,
   or an internally-trained debiased classifier from the literature.
3. **IRB / ethics review** not documented. Generated faces are not real
   people, but FairFace seeds are real photos. Confirm the institutional
   position before public release.
4. **License** still TBD in `README.md` line 152. Pick MIT or Apache 2.0
   before the repo flips public at submission.

## 3. Architecture review

### What is right

- **Layered separation.** `generator/`, `validator/`, `auditors/`,
  `analysis/`, `viz/` are clean modules with no upward dependencies. A
  P2/P3/P4 contributor can rerun any single layer without touching the
  others. This is exactly the reusable-instrument property the master
  plan asks for.
- **Auditor registry pattern** (`src/cap/auditors/registry.py`,
  `base.py`) makes adding or swapping auditees a config-only change.
  This is what "one pipeline → multiple papers" looks like in code.
- **Manifest-based reproducibility.** `RunManifest` captures git hash,
  resolved config, seeds, model hashes — the seven-item reproducibility
  list in `README.md` is honored. Critical for ISR and for the P4
  longitudinal contrast.
- **Streaming manifest writes to local disk + sync to Volume**
  (`53a1d3a`, `434e67e`) is a sensible Volume-FUSE workaround that
  preserves crash safety without compromising the manifest's role as
  the single source of truth.
- **Idempotency in the Ray runner** (`9a63196`) — restarts don't double-
  generate. Important when L4 sessions can drop mid-run.
- **Pose-keypoint ControlNet** correctly chosen over canny/depth. The
  reasoning in `CLAUDE.md` item 1 is exactly the right argument for an
  ISR reviewer: edges encode demographic-correlated features, so locking
  them creates a confound. This will survive review.
- **Quality-first priority ordering** in `CLAUDE.md` (`quality >
  reproducibility > cost > speed > convenience`) is the correct posture
  for an A* submission and matches the master plan's stance.

### What is missing or weak

- **`src/cap/viz/` is thin.** Only `cross_system.py`, `regression.py`,
  `theme.py` are present. The implementation plan specifies 5 figures
  for the paper (intersectional heatmap, flip-rate forest plot,
  ordinal-logit dose-response, ArcFace ID similarity histogram,
  cross-system contrast). Currently only the intersectional heatmap is
  visible in `results/mvp_demo/viz/figures/`. Building the remaining
  four figures should happen before scaling, so the full-run output
  drops directly into the paper.
- **No `tests/` for the analysis module.** `tests/` exists but I did
  not see coverage of `statistical_tests.py`. The hypothesis-test code
  is exactly what reviewer-2 will scrutinize. A few small tests (e.g.
  McNemar's on a synthetic table with known p-value, ordinal logit on
  a monotonic synthetic dose-response) would be cheap insurance.
- **No `system_signature` field in `RunManifest`** for auditor
  responses. Auditee APIs change silently; for P4 to be credible we
  need to be able to prove "the API contract was X at T0 and Y at T1".
  Recommend adding a hash of the full first response schema per auditor.
- **No negative-control auditor** registered. See §2 master-plan gap 2.
- **No "smoke vs. production" gate in code.** `CLAUDE.md` item 5 says
  smoke notebooks may take quality shortcuts and must never be used as
  the production code path — but this is enforced only by convention.
  A simple assertion in the production CLI (e.g. refuse to run if
  config matches the smoke profile) would harden it.

## 4. MVP results — the identity-preservation failure

### What the numbers say

```json
"identity": {
  "mean_cosine_sim": -0.0036512202930485244,
  "median_cosine_sim": -0.006570568308234215,
  "preserved_fraction": 0.0,
  "n": 600
}
```

ArcFace embeddings are L2-normalized 512-dim vectors. Cosine similarity
between *the same* identity across pose/expression typically lands in
0.5–0.85; cosine sim between *different* identities is ~0. Our 600
generated images have a *median* cosine sim to their seed of −0.0066, and
not a single image clears the 0.5 threshold.

In plain English: across 50 identities and 12 counterfactual variations
each, the generator produced 600 images that, by ArcFace's measure, are
**unrelated** to the seeds.

### What this implies for the headline result

The MVP also reports:

```json
"flip_rates": [
  {"auditor": "deepface", "task": "age",    "coarse_flip_rate": 1.0, "n_identities": 50},
  {"auditor": "deepface", "task": "gender", "coarse_flip_rate": 1.0, "n_identities": 50}
],
"pairwise_flip_rates": [
  {"task": "age",    "mean_pairwise_flip_rate": 0.85, ...},
  {"task": "gender", "mean_pairwise_flip_rate": 0.54, ...}
]
```

A coarse flip rate of 1.0 means every single seed had at least one
counterfactual that flipped DeepFace's prediction. Combined with
`preserved_fraction = 0.0`, the most parsimonious explanation is *not*
"DeepFace is exquisitely sensitive to demographic counterfactuals" —
it's "the counterfactuals are different people, so of course the
prediction changes."

The pairwise flip rate (0.54 for gender, 0.85 for age) is what you'd
expect from comparing essentially random face pairs through a
classifier — gender is binary so two random faces flip ~50% of the
time, age is multi-class so they flip much more often.

The MVP signal, taken at face value, is consistent with a generator
that has lost identity conditioning entirely. We cannot distinguish
this from "the auditor is biased" using the MVP outputs.

### What the pre-registration says

From `drafts/isr-counterfactual-audit/preregistration.md`:

> H5 — ArcFace identity preservation ≥ 80% (manipulation check, blocks
> main analyses if fails)

`preserved_fraction = 0.0` is not a marginal failure. Filing this run
under the current pre-registration would force us to report H5 as
falsified and decline to test H1–H4. That is the *correct* behavior of
a pre-registration — it is doing its job.

This is also why **the pre-registration must not be filed yet.** Filing
before identity preservation is verified would commit us to publishing
a paper whose first reported result is "we built a pipeline that doesn't
preserve identity." Fix first, file second.

### Likely root causes (ordered by my prior)

1. **PuLID embedding silently not injected into the Flux pipeline.**
   The most common failure mode for PuLID-Flux is that the identity
   embedding is computed and passed to the pipeline call, but the
   pipeline silently ignores it because the PuLID hooks weren't
   registered onto the transformer blocks. This produces Flux output
   that is text-prompt-conditioned only — visually plausible faces, but
   identity comes from the prompt, not the seed. Symptoms match
   exactly: `preserved_fraction = 0.0`, mean cosine sim ≈ 0, every
   counterfactual looks like a different person.
   **How to verify:** generate one image with PuLID embedding for seed
   A and one with embedding for seed B, identical prompt, identical
   noise seed. If the two outputs are nearly identical, PuLID hooks
   are not wired in.

2. **ArcFace embedding computed on misaligned crops.** If the validator
   feeds the seed image and the generated image to ArcFace without
   running antelopev2's face detector + alignment first (or runs them
   with different alignments), embeddings will be near-orthogonal even
   for the same identity. Cosine sim distribution would look noisy,
   centered near 0.
   **How to verify:** compute embedding(seed) vs embedding(seed) — must
   return ~1.0. Then embedding(seed) vs embedding(seed crop with 5%
   padding change) — should still be > 0.6. If either fails, alignment
   is the bug.

3. **Wrong pair being scored.** If the validator is comparing
   `embedding(generated_i) vs embedding(generated_j)` instead of
   `embedding(seed) vs embedding(generated_i)`, you'd see cosine sims
   centered near 0 even with a healthy generator (different
   counterfactuals are by design different people on the
   gender/skin-tone axes).
   **How to verify:** read `src/cap/validator/identity.py` and confirm
   the seed image is loaded and embedded as the reference, not a
   sibling generated image.

4. **antelopev2 model loaded but pointed at the wrong weights.** Less
   likely given commit `d021d53` (`ArcFaceIdentityValidator: accept
   root + pre-stage antelopev2`) explicitly addressed model staging,
   but worth confirming the loaded weights match what was staged.

I rank (1) highest because the failure is so clean (n = 600,
preserved = 0). Misalignment or wrong-pair bugs typically produce a
*distribution* of cosine sims, including some accidental hits. A flat
preserved_fraction = 0.0 across 600 images is the signature of "PuLID
isn't doing anything."

### What to do before generating again

1. Add a unit test: load FairFace seed, embed it, embed it again with
   trivial augmentation, assert cosine sim > 0.9. Catches alignment bugs.
2. Add a sanity assertion in the generator: render two images with the
   same prompt, same noise, but PuLID embeddings from different seeds.
   Assert the resulting ArcFace embeddings are *not* near-identical
   (sim < 0.95). Catches the silent-PuLID-bypass failure.
3. Run a 10-seed × 4-counterfactual mini-MVP. If preserved_fraction
   exceeds 0.8 there, scale to the 50 × 12 again. If it doesn't, do
   not scale.

Only after preserved_fraction > 0.8 on a re-run should the OSF pre-
registration be filed and full-run compute be authorized.

## 5. Storage and publication strategy

The pivot from GCS to **Databricks Volume + HuggingFace Datasets +
Zenodo embargoed DOI** is an *improvement* over the original plan, not a
drift.

Why it's better:
- **Reviewers see a registered DOI from submission.** Zenodo's
  embargoed-access flow is the canonical IS pattern (ISR / MISQ /
  JMIS / JAIS all accept it). The DOI exists and is citable from the
  submitted manuscript; the data unlocks on acceptance.
- **URL-stable private→public flip.** `alenjani/cap-counterfactuals`
  on HF stays the same string forever, so the manuscript URL doesn't
  rot.
- **`load_dataset()` one-liner.** Downstream researchers in P2–P4 (and
  third parties) get a frictionless interface that GCS cannot match.
- **No standing GCS bill across the project lifetime.** The Volume +
  HF model cache means Flux's 24 GB downloads once across all clusters.

The legacy `gcp/` and `configs/gcp.yaml` directories are correctly
flagged in `README.md` as predating this decision. Recommend either
removing them or moving them to `legacy/` to prevent confusion at
submission time.

## 6. Engineering principles in CLAUDE.md

`counterfactual-audit-pipeline/CLAUDE.md` is excellent. The six
quality-non-negotiable items match the ISR review surface exactly:

- **Item 1 (pose ControlNet, not canny)** — defends against the
  edge-confound objection that any face-fairness reviewer will raise.
- **Item 2 (PuLID required)** — defends against "you used a weaker
  identity model than SOTA" — though see §4: this principle must
  actually hold in execution, not just in policy.
- **Item 3 (FP8 floor, NF4 only when forced)** — `mvp.yaml` documents
  why NF4 was chosen (FP8 OOMs at 1024² on L4). This trace is exactly
  what the paper's reflective account section should reproduce.
- **Item 4 (768² floor, 1024² actual)** — survives reviewer scrutiny.
- **Item 5 (smoke ≠ production)** — currently a convention; recommend
  hardening to an assertion in the production CLI (see §3).
- **Item 6 (manifest-based reproducibility)** — wired and honored.

The priority ordering `quality > reproducibility > cost > speed >
convenience` is the right stance for A* venue submissions.

## 8. Bottom line

The repository is a faithful realization of the four-paper research
program's reusable-instrument design. Architecture, statistical
plumbing, and storage strategy are all where they should be. The
quality principles in `CLAUDE.md` will survive A* review.

But **the MVP did not actually test what we claim it tested.** With
preserved_fraction = 0.0, every downstream number in the MVP summary
is a measurement of "Flux generates faces from text" rather than
"counterfactual demographic perturbation flips classifier output."

The fix is local — almost certainly in PuLID hook registration, the
ArcFace alignment path, or the validator's pair selection. Until it
lands and verifies on a small re-run, do not file the pre-registration
and do not scale to the full audit.

When that fix lands, schedule a Review 002 against the post-fix MVP
and the proposed full-run config.
